<objective>
## Goal
Add login page UI and wire admin authentication into the reading list page, enabling admin-mode controls (status writes, edit/delete buttons on cards, FAB toggle).

## Purpose
Admin functionality is the bridge between the public reading list (Phase 4) and full content management. Without login UI and admin-mode, there's no way to manage links from the browser.

## Output
- SSR login page at `/login` with email/password form
- Admin auth integration in `reading-list.js`: FAB button, session check, admin-mode class toggle, status PATCH writes, edit/delete card buttons
- Standalone `admin-auth.js` for the login page form handling
</objective>

<task type="auto">
  <name>Task 2: Wire admin auth into reading-list.js</name>
  <files>public/reading-list.js</files>
  <action>
    Add admin functionality to reading-list.js (reference original reading-list.js lines 700-882):

    1. Session check on page load:
       - GET /api/auth/session
       - If authenticated: set state.isAdmin = true, call enterAdminMode()
       - If not: remain in public mode

    2. FAB button behavior:
       - Already exists in SSR shell from Phase 4
       - When not admin: click navigates to /login?redirect=/reading-list
       - When admin: click calls logout (POST /api/auth/logout), then exitAdminMode()

    3. enterAdminMode():
       - document.body.classList.add('admin-mode')
       - Update FAB icon to lock-open SVG
       - Show admin-add-btn in filter bar

    4. exitAdminMode():
       - document.body.classList.remove('admin-mode')
       - Update FAB icon to lock SVG
       - Hide admin-add-btn
       - state.isAdmin = false

    5. Edit/Delete buttons on cards:
       - In buildCard(), add edit button (pencil icon) and delete button (trash icon)
       - Wrap in a div with class "card-admin-actions"
       - These are hidden by default CSS, shown when body has admin-mode class
       - Edit button: onclick opens edit modal (stub with TODO for Plan 05-02)
       - Delete button: onclick calls deleteLink(id) with confirm()

    6. Status popover writes (unwire the TODO stub):
       - When admin clicks a status in the popover:
         PUT /api/links/:id with { status: newStatus }
       - On success: update local state, re-render card, show toast
       - On failure: show error toast
       - When not admin: status popover remains read-only (no writes)

    7. deleteLink(id):
       - confirm("Delete this link?")
       - DELETE /api/links/:id
       - On success: remove from state.allLinks, re-render, show toast
       - On failure: show error toast

    CSS additions needed in the SSR shell (pages.tsx):
    - .card-admin-actions { display: none; }
    - body.admin-mode .card-admin-actions { display: flex; }
    - .admin-add-btn { display: none; }
    - body.admin-mode .admin-add-btn { display: inline-flex; }
  </action>
  <verify>npx tsc --noEmit passes; admin login activates admin-mode; status writes persist; delete removes link</verify>
  <done>AC-3, AC-4, AC-5 satisfied: Admin mode activates, status writes work, logout works</done>
