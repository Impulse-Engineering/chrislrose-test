  <name>Task 1: Write queries + API endpoints for gear, categories, and content</name>
  <files>src/db/queries.ts, src/routes/api/gear.ts, src/routes/api/categories.ts, src/routes/api/content.ts, src/index.ts</files>
  <action>
    **1a. Write queries in queries.ts:**
    Add CRUD query functions for gear tables. Use a generic pattern since all 5 gear tables have similar structure:

    - `createGearItem(db, table, data)` — INSERT into gear_{table}, return created item
    - `updateGearItem(db, table, id, data)` — UPDATE gear_{table} WHERE id=?, return updated item
    - `deleteGearItem(db, table, id)` — DELETE FROM gear_{table} WHERE id=?, return boolean
    - `replaceCategories(db, categories: {name: string, sort_order: number}[])` — DELETE all rows, INSERT new list (transaction)
    - `createCategory(db, name, sort_order)` — INSERT into categories
    - `deleteCategory(db, name)` — DELETE FROM categories WHERE name=?
    - `updateSiteContent(db, id, content)` — UPDATE site_content SET content=?, updated_at=? WHERE id=?

    Table name validation: only allow 'hardware', 'software', 'hobbies', 'projects', 'podcasts' — prevent SQL injection.

    **1b. Gear write API endpoints in gear.ts:**
    Add protected routes (using requireAuth middleware):
    - POST /api/gear/:type — create gear item
    - PUT /api/gear/:type/:id — update gear item
    - DELETE /api/gear/:type/:id — delete gear item
    Validate :type against allowed table names. Return 400 for invalid type.

    **1c. Categories write API in categories.ts:**
    Add protected route:
    - POST /api/categories — accepts { categories: [{name, sort_order}] }, replaces all categories

    **1d. Content API (new file src/routes/api/content.ts):**
    - PUT /api/content/:id — protected, updates site_content row

    **1e. Register content route in index.ts:**
    Import contentRoutes and mount at /api/content

    Avoid: Dynamic table names in SQL without validation. Always validate table name against an allow list before interpolating.
  </action>

  <done>AC-1, AC-2, AC-3, AC-4, AC-7 satisfied</done>

  <action>
    **2a. SSR admin page in pages.tsx:**
    Add GET /admin route (auth-gated: check session, redirect to /login?redirect=/admin if not authenticated).

    Page structure (match original admin.html layout):
    - Section header: "// admin" eyebrow, "Admin Dashboard" h2
    - Tab bar: "Reading List", "My Gear", "Now", plus "Sign out" button
    - Reading List tab: category chips container, add input + button, status div
    - My Gear tab: 5 subsections (Hardware, Software, Projects, Hobbies, Podcasts) each with title, add button, list container
    - Now tab: description text, textarea, updated-at display, save button + status
    - Gear modals: one modal per type (hw, sw, hb, pr, pod) — each with type-specific fields matching original admin.html
    - Link to admin.js script

    **2b. Client-side admin.js:**
    Port the original admin.js (~600 lines) to use fetch-based API instead of Supabase. Key adaptations:

    - Auth: No Supabase auth. Session already handled by cookie. Check GET /api/auth/session on load → if not authenticated, redirect to /login?redirect=/admin
    - Tab switching: querySelectorAll('.admin-tab') click handlers toggle .active on tabs and sections
    - loadCategories(): GET /api/categories → render chips with delete buttons (createElement, NOT innerHTML)
    - saveCategories(): POST /api/categories with sorted list
    - loadGear(): GET /api/gear/hardware etc → render lists with edit/delete buttons (createElement)
    - Gear modals: open/close/save using fetch POST/PUT to /api/gear/:type
    - Delete gear: fetch DELETE /api/gear/:type/:id
    - loadNow(): GET /api/gear/now → populate textarea
    - saveNow(): PUT /api/content/now
    - Logout: POST /api/auth/logout → redirect to /login

    Use vanilla JS (var, function declarations, .then() chains — match existing code style).
    Use textContent and createElement — NO innerHTML (security hook compliance).
    Generate IDs client-side: type prefix + hash + timestamp (e.g., hw_abc123_xyz).

    Avoid: ES6+ syntax (arrow functions, const/let, template literals). Match existing codebase style.
