## Goal
Create Links CRUD API and Categories API endpoints backed by D1, with typed query helpers. Public endpoints (GET) are accessible without auth; write endpoints (POST/PUT/DELETE) require admin auth.

## Purpose
The reading list is the core feature of the site. The API layer provides data for the client-side reading list page (Plan 02) and will be reused by the admin system (Phase 5).

## Output
- Links API: GET /api/links, GET /api/links/:id, POST /api/links, PUT /api/links/:id, DELETE /api/links/:id
- Categories API: GET /api/categories
- D1 query functions for links and categories in queries.ts
</objective>

<task type="auto">
  <name>Task 1: D1 query functions for links and categories</name>
  <files>src/db/queries.ts</files>
  <action>
    Add typed query functions to the existing queries.ts:

    **Links queries:**
    - `getLinks(db, opts?)` — SELECT * FROM links, with optional filters:
      - `category?: string` — WHERE category = ?
      - `status?: string` — WHERE status = ?
      - `includePrivate?: boolean` — if false/undefined, add WHERE private = 0
      - Always ORDER BY saved_at DESC
    - `getLinkById(db, id)` — SELECT * WHERE id = ?
    - `createLink(db, data)` — INSERT with all fields. Generate `id` as `Date.now().toString(36)`. Set `saved_at` to ISO string. Set `read` to 1 if status is 'done', else 0.
    - `updateLink(db, id, data)` — UPDATE all mutable fields. Keep `read` in sync with `status`.
    - `deleteLink(db, id)` — DELETE WHERE id = ?

    **Categories queries:**
    - `getCategories(db)` — SELECT name, sort_order FROM categories ORDER BY sort_order ASC

    Use the existing Link and Category types from types.ts. Follow the same pattern as existing admin/session queries (prepare/bind/first or all).
  </action>
  <verify>npx tsc --noEmit passes with no errors</verify>
  <done>AC-6 satisfied: TypeScript compiles. Query functions ready for API routes.</done>
</task>

<task type="auto">
  <name>Task 2: Links API route + Categories API route + mounting</name>
  <files>src/routes/api/links.ts, src/routes/api/categories.ts, src/index.ts</files>
  <action>
    **Create src/routes/api/links.ts:**
    - Hono router with Env bindings
    - GET / — call getLinks with query params (category, status from c.req.query). Always pass includePrivate: false for public. Return JSON array.
    - GET /:id — call getLinkById, return 404 if not found, return JSON.
    - POST / — parse JSON body, call createLink, return 201 with created link.
    - PUT /:id — parse JSON body, call updateLink, return 200 with updated link. Return 404 if link not found.
    - DELETE /:id — call deleteLink, return 204.

    **Create src/routes/api/categories.ts:**
    - Hono router with Env bindings
    - GET / — call getCategories, return JSON array.

    **Update src/index.ts:**
    - Import linkRoutes and categoryRoutes
    - Mount GET routes BEFORE requireAuth middleware (public access), same pattern as gear API:
      ```
      app.route('/api/links', linkRoutes);      // public GET, protected POST/PUT/DELETE
      app.route('/api/categories', categoryRoutes); // public GET
      ```
    - The existing `app.use('/api/*', requireAuth)` will protect POST/PUT/DELETE on links automatically since GET routes are defined on the sub-router but writes go through the middleware.

    **Important:** The Hono sub-router approach means ALL methods on `/api/links` match the route. To make GET public but POST/PUT/DELETE protected:
    - Mount the links route BEFORE the requireAuth middleware (same as gear API pattern)
    - In the links router, explicitly apply requireAuth to write methods:
      ```typescript
      import { requireAuth } from '../../middleware/auth';
      links.post('/', requireAuth, async (c) => { ... });
      links.put('/:id', requireAuth, async (c) => { ... });
      links.delete('/:id', requireAuth, async (c) => { ... });
      ```
    - This keeps GET public while protecting writes.
  </action>
